# План: расширение ИИ-помощника и голосовой ввод

Документ продолжает [llm-vector-search.md](./llm-vector-search.md). Предполагается, что MVP уже работает: **embeddings + pgvector → список мест → подсветка на карте**.

## Цель

Посетитель может **говорить** в микрофон — *«где круассан»*, *«покажи туалет»*, *«что рядом с Zara»* — и получать тот же результат на карте, плюс при необходимости **короткий диалог** с уточнениями.

Помощник по-прежнему **не знает мир** — только зоны и объекты ТЦ с тегами. Голос — это способ ввода; «умность» — в retrieval и (опционально) в маленькой LLM для переформулировки и уточнений.

**Не в scope:** голосовой ассистент «как Siri» с общими знаниями, построение маршрута по графу коридоров (отдельная задача), распознавание лиц.

---

## Упрощённый сценарий (рекомендуемый MVP голоса)

```
🎤 «блин где майка»
        │
        ▼
  STT (речь → текст)          ← новый слой
        │
        ▼
  POST /assistant/find        ← уже есть / планируется
        │
        ▼
  список зон + подсветка
        │
        ▼
  (опционально) TTS: «Нашёл 3 места»
```

**Для голосового MVP LLM не нужна.** Достаточно STT + существующий семантический поиск.

| Что делает «ИИ» | Как |
|---------------|-----|
| Понимает произнесённый запрос | STT (Whisper / Web Speech API) |
| Находит места по смыслу | Embeddings + pgvector (как в MVP) |
| Не выдумывает магазины | Результат только из БД |
| Отвечает голосом | TTS (опционально, шаблонные фразы) |

Текст на экране — тот же шаблон: *«Нашли 4 места»* + карточки. Голос — дублирует короткую фразу, не генерирует эссе.

### Когда понадобится больше, чем STT + find (уровень 2+)

| Запрос | STT + find | + LLM / логика |
|--------|------------|----------------|
| *«майка»* (голосом) | ✅ | избыточно |
| *«а подешевле?»* (уточнение) | ❌ | нужна история + LLM или фильтры |
| *«что рядом с Zara?»* | ❌ | гео-логика или LLM + координаты |
| *«покажи на втором этаже»* | ⚠️ частично (фильтр этажа) | LLM извлекает intent или regex |
| *«нет, не то, кофе»* (исправление) | ❌ | контекст диалога |

**Вывод:** голос добавляет **один слой (STT)** к уже спланированному MVP. Расширенный ассистент — это **уровень 2** из исходного плана, плюс опционально TTS и гео-запросы.

## Пользовательские сценарии

### Посетитель — голос (MVP)

1. На карте нажимает кнопку 🎤 (или удерживает «говорить»).
2. Говорит: *«где круассан»*, *«туалет»*, *«H&M»*.
3. Видит распознанный текст (для доверия и исправления опечаток STT).
4. Получает список мест + подсветку на карте.
5. При ошибке распознавания — может допечатать вручную в той же строке.

### Посетитель — расширенный ассистент (уровень 2)

1. Ведёт короткий диалог: *«где поесть»* → *«что-нибудь быстрое»* → список фуд-кортов.
2. Уточняет: *«покажи только второй этаж»* — фильтр по `floor_label`.
3. Спрашивает: *«что рядом с Zara?»* — зоны в радиусе N метров на том же этаже.
4. Слышит краткий голосовой ответ (TTS): *«Нашёл три места на втором этаже»*.

### Киоск в ТЦ

1. Большая кнопка «Спросить», always-on микрофон **не** нужен (шум зала).
2. Push-to-talk: нажал → говоришь → отпустил → поиск.
3. Таймаут тишины 2–3 с автоматически завершает запись.
4. Громкий динамик или наушники для TTS (опционально).

### Админ

Без изменений относительно [llm-vector-search.md](./llm-vector-search.md): теги, описание, переиндексация при сохранении.

## Архитектура голосового MVP (уровень 2)

```
┌─────────────────┐                              ┌──────────────┐
│  Vue            │  audio blob / Web Speech     │  FastAPI     │
│  🎤 кнопка      │ ───────────────────────────► │  (опционально│
│  Assistant UI   │     или STT в браузере       │   /stt)      │
└────────┬────────┘                              └──────┬───────┘
         │                                                │
         │  query text                                    │ Whisper API
         ▼                                                ▼
┌─────────────────┐    POST /assistant/find     ┌──────────────┐
│  распознанный   │ ──────────────────────────► │  embed +     │
│  текст + карта  │ ◄────────────────────────── │  pgvector    │
└─────────────────┘   zones[]                   └──────────────┘
```

### Два варианта STT

#### Вариант A — в браузере (Web Speech API)

```
Микрофон → SpeechRecognition (Chrome) → текст → /assistant/find
```

**Плюсы:** $0, нет загрузки аудио на сервер, latency низкая.  
**Минусы:** Safari/Firefox слабее; качество RU зависит от ОС; на киоске нужен Chrome; нет единого API в production без полифиллов.

#### Вариант B — серверный Whisper (рекомендуется для prod)

```
Микрофон → MediaRecorder → POST /assistant/transcribe → текст → /assistant/find
```

**Плюсы:** стабильное качество RU, одинаково на всех браузерах, можно `whisper-small` локально.  
**Минусы:** +200–800 ms, нужен endpoint, при облаке — плата за минуту.

#### Вариант C — гибрид (рекомендуется)

1. **По умолчанию** — Web Speech API (быстрый feedback в Chrome).
2. **Fallback** — серверный Whisper, если браузер не поддерживает или confidence низкий.
3. Пользователь всегда видит текст до отправки в поиск (исправить одним тапом).

### Поток запроса (голос)

1. Пользователь удерживает 🎤, говорит: *«где найти круассан»*.
2. STT → текст `"где найти круассан"` (показать в UI).
3. Тот же путь, что текстовый MVP: embed → pgvector → зоны.
4. Карта подсвечивает результаты.
5. (Опционально) TTS: `speechSynthesis.speak("Нашёл два места")` — встроено в браузер, без сервера.

### Архитектура уровня 3 (диалог + голос)

```
┌──────────┐   STT    ┌─────────────┐   intent + slots   ┌─────────┐
│  🎤 UI   │ ───────► │  /assistant │ ─────────────────► │  LLM    │
│  + чат   │ ◄─────── │  /chat      │ ◄───────────────── │  3B     │
└──────────┘   TTS    └──────┬──────┘   tool: find_zones  └─────────┘
                             │
                             ▼
                      pgvector + geo filter
```

LLM **не ищет магазины сама** — вызывает tool `find_zones(query)` и `filter_nearby(zone_id, radius)`. Ответ строится только из tool results (как в исходном плане).

## Расширения ассистента (помимо голоса)

### 1. Поиск по объектам сцены (POI)

Уже есть `scene_objects` с тегами (`туалет`, `лифт`, `банкомат`). Индексировать **и зоны, и объекты** в одной таблице или двух с общим search endpoint.

```sql
search_index (
  mall_id, floor_id, entity_type,  -- 'zone' | 'scene_object'
  external_id, name, tags, embedding, position
)
```

Ответ: `{ type: "scene_object", name: "Туалет", floor_label: "1 этаж", position: [x, z] }` → `flyTo` на точку, не полигон.

### 2. Фильтры без LLM

| Intent | Реализация |
|--------|------------|
| *«на втором этаже»* | Парсинг `этаж` / `2 этаж` + filter `floor_label` |
| *«только кафе»* | filter `category = cafe` если слово в whitelist |
| *«ближе ко мне»* | Нужна геолокация киоска / QR у входа → sort by distance |

Regex + словарь intents покрывает 60–70% уточнений **без LLM**.

### 3. «Рядом с X»

1. Найти зону X по имени (exact / fuzzy / embedding).
2. Взять центр полигона зоны или `position` объекта.
3. Выбрать другие зоны на том же `floor_id` в радиусе R (евклид по xz).
4. Исключить саму зону X.

LLM может только переформулировать: *«Рядом с Zara: Pull&Bear, Bershka…»*.

### 4. История диалога (уровень 3)

```sql
assistant_sessions (
  id uuid, mall_id, created_at, channel  -- 'text' | 'voice'
)
assistant_messages (
  session_id, role, content, zones_json, created_at
)
```

Хранить 5–10 последних реплик для контекста LLM. Для киоска — сброс сессии через 60 с неактивности (приватность).

### 5. Аналитика запросов

Логировать: `query_text`, `source` (text/voice), `stt_confidence`, `top_zone_ids`, `latency_ms`. Без хранения аудио по умолчанию (GDPR / 152-ФЗ).

### 6. Мультиязычность

| Слой | Подход |
|------|--------|
| STT | Whisper multilingual; Web Speech `lang: ru-RU` |
| Embeddings | `bge-m3` (уже в плане) |
| UI | i18n Vue; ответы-шаблоны на ru/en |
| LLM | `qwen2.5` / GPT с system prompt «отвечай на языке запроса» |

## Выбор технологий: голос

### Speech-to-Text (STT)

| Вариант | Качество RU | Latency | Стоимость | Офлайн | Комментарий |
|---------|-------------|---------|-----------|--------|-------------|
| **Web Speech API** | Среднее–хорошее (Chrome) | 100–300 ms | $0 | Частично | Быстрый старт, не для всех браузеров |
| **OpenAI Whisper API** | Хорошее | 300–800 ms | ~$0.006/мин | Нет | Простой prod fallback |
| **Whisper локально** (`small` / `base`) | Хорошее | 500 ms–2 s (CPU) | $0 | Да | Киоск / VPS без GPU: `small` |
| **faster-whisper** | То же | 2–4× быстрее | $0 | Да | Рекомендуется для self-host |
| Yandex SpeechKit | Отличное RU | 200–500 ms | Платно | Нет | Если уже в экосистеме Яндекса |
| Google Cloud STT | Хорошее | 200–500 ms | Платно | Нет | Vendor lock-in |

**Рекомендация:**

- **Пилот:** Web Speech API в Chrome + ручная правка текста.
- **Prod / киоск:** `POST /assistant/transcribe` на **faster-whisper `small`** (CPU) или OpenAI Whisper API.
- **Максимальное RU:** Yandex SpeechKit (если бюджет есть).

### Text-to-Speech (TTS, опционально)

| Вариант | Качество | Стоимость | Комментарий |
|---------|----------|-----------|-------------|
| **Web Speech API** `speechSynthesis` | Среднее | $0 | Хватит для «Нашёл 3 места» |
| OpenAI TTS | Хорошее | ~$15/1M chars | Избыточно для коротких фраз |
| Silero TTS | Хорошее RU | $0 локально | ~50 MB модель, офлайн киоск |
| Yandex SpeechKit TTS | Отличное RU | Платно | Для премиум-киоска |

**Рекомендация:** шаблонные фразы через `speechSynthesis`; Silero — если киоск без интернета и нужен приятный голос.

### Формат аудио с клиента

| Параметр | Значение |
|----------|----------|
| Кодек | `webm/opus` (MediaRecorder default) или `wav` после конвертации |
| Длительность | max 15–20 с на запрос |
| Sample rate | 16 kHz достаточно для Whisper |
| Max размер | ~500 KB → лимит в nginx / FastAPI |

## API

### Голосовой MVP

```
POST /api/malls/{mall_id}/assistant/transcribe
  Content-Type: multipart/form-data
  body: audio file (webm/wav), optional lang=ru
  response: {
    "text": "где круассан",
    "confidence": 0.92,
    "duration_ms": 1800
  }
```

Затем клиент вызывает существующий:

```
POST /api/malls/{mall_id}/assistant/find
  body: { "query": "где круассан", "limit": 10, "source": "voice" }
```

Или **объединённый** endpoint (меньше round-trips):

```
POST /api/malls/{mall_id}/assistant/find-voice
  body: audio file
  response: {
    "transcript": "где круассан",
    "zones": [ ... ]
  }
```

### Уровень 3 — диалог

```
POST /api/malls/{mall_id}/assistant/chat
  body: {
    "session_id": "uuid | null",
    "message": "а подешевле?",
    "source": "voice" | "text"
  }
  response: {
    "session_id": "...",
    "message": "Вот более бюджетные варианты...",
    "zones": [ ... ],
    "actions": [{ "type": "highlight" }, { "type": "fly_to", "zone_id": "..." }]
  }
```

### Гео-запрос

```
POST /api/malls/{mall_id}/assistant/nearby
  body: {
    "anchor_query": "Zara",      -- или zone_id
    "query": "кофе",             -- опционально: фильтр по смыслу
    "radius_m": 50,
    "limit": 5
  }
```

`/transcribe` и `/find` — публичные, rate limit строже чем у текстового поиска (аудио тяжелее).

## Изменения во frontend

### 1. `AssistantSearch.vue` — режим ввода

- Строка текста (как в MVP).
- Кнопка 🎤 push-to-talk (`mousedown` / `touchstart` → запись, `mouseup` → стоп).
- Индикатор записи (пульс / waveform опционально).
- Блок «Вы сказали: …» с возможностью редактирования перед поиском.
- Состояния: `idle` | `listening` | `transcribing` | `searching` | `results`.

### 2. Composable `useVoiceInput.js`

```javascript
// псевдокод
const { isListening, transcript, start, stop, error } = useVoiceInput({
  mode: 'auto',  // 'webspeech' | 'server' | 'auto'
  lang: 'ru-RU',
  onTranscript: (text) => assistant.find(text),
})
```

- `auto`: Web Speech если `webkitSpeechRecognition`, иначе MediaRecorder → `/transcribe`.
- Обработка `NotAllowedError` (микрофон запрещён) — подсказка в UI.

### 3. Composable `useAssistant.js` (расширение)

- `find(query, { source })` — как в MVP.
- `findVoice(audioBlob)` — transcribe + find.
- `chat(sessionId, message)` — уровень 3.
- Debounce только для текста; для голоса — по событию «конец записи».

### 4. TTS `useVoiceOutput.js`

```javascript
speak('Нашли 3 места на первом этаже', { lang: 'ru-RU' })
```

Отключение звука в настройках киоска. Не говорить поверх нового запроса — `cancel()` перед следующим `speak`.

### 5. Киоск-режим

- Крупные touch-targets (мин. 48×48 px).
- `prefers-reduced-motion` — без лишней анимации.
- Авто-сброс результатов через N секунд.

**Объём:** голосовой MVP ~2–3 дня поверх текстового MVP; диалог + TTS + geo ~+4–5 дней.

## Оценка ресурсов и стоимости

### Дополнительная нагрузка (голос)

| Параметр | Оценка |
|----------|--------|
| Средняя длина фразы | 3–5 с |
| Запросов голосом в день | 5–50% от текстовых (зависит от киоска) |
| Размер аудио на запрос | 30–100 KB |

### STT: стоимость

| Сценарий | Расчёт | Сумма |
|----------|--------|-------|
| Web Speech only | — | **$0** |
| OpenAI Whisper | 100 голосовых/день × 5 с × 30 дней ≈ 250 мин/мес | **~$1.5/мес** |
| faster-whisper на VPS | Тот же CPU что embeddings | **$0** marginal |
| Yandex SpeechKit | ~₽0.16/15 с | **~₽500–2000/мес** при активном киоске |

### Latency (end-to-end, голос)

| Этап | Web Speech | Server Whisper |
|------|------------|----------------|
| STT | 100–300 ms | 300–800 ms |
| embed + pgvector | 100–500 ms | 100–500 ms |
| **Итого** | **200–800 ms** | **400–1300 ms** |

Приемлемо для киоска: показывать спиннер на этапе «Распознаём…».

### Железо для локального Whisper

| Модель | RAM | RTF на CPU | Комментарий |
|--------|-----|------------|-------------|
| `tiny` | ~1 GB | ~0.3× | Слабое качество RU |
| **`small`** | ~2 GB | ~0.5–1× | **Баланс для киоска** |
| `medium` | ~5 GB | ~2× | Нужен GPU или терпение |

На том же мини-ПК что Ollama embeddings: **small** на CPU — реально для фраз 3–5 с.

### TTS

| Вариант | Стоимость |
|---------|-----------|
| `speechSynthesis` | $0 |
| Silero локально | $0, +~50 MB диск |

```
┌─────────────────────────────────────┐
│  Киоск (Chrome / Electron)          │
│  Vue + Web Speech (fallback)        │
└──────────────┬──────────────────────┘
               │ если нет сети → локальный API
               ▼
┌─────────────────────────────────────┐
│  FastAPI на мини-ПК                 │
│  ├── faster-whisper small (STT)     │
│  ├── Ollama bge-m3 (embed)          │
│  └── (опц.) qwen2.5:3b (chat)       │
└──────────────┬──────────────────────┘
               ▼
        Supabase / локальный Postgres + pgvector
```

**Диск:** +~500 MB к MVP (Whisper small). **RAM:** 8 GB комфортно для embed + whisper sequential; 16 GB если одновременно LLM.

## Риски и как снизить

| Риск | Митигация |
|------|-----------|
| Шум в зале ТЦ | Push-to-talk, не always-listening; noise gate в UI «держите ближе» |
| STT путает бренды (*«Зара»* / *«зала»*) | Показать transcript; fuzzy match по `name` в retrieval |
| Пользователь не дал микрофон | Текстовый ввод всегда доступен |
| Safari без Web Speech | Авто-fallback на серверный STT |
| Хранение аудио / приватность | Не хранить аудио; логировать только текст; TTL на сессии |
| LLM выдумывает магазины | Tool calling только; system prompt «только из zones[]» |
| Долгий ответ блокирует очередь киоска | Таймаут 10 с; кнопка «Отмена» |

---

## Рекомендуемый путь

1. **Сначала** довести текстовый MVP (embeddings + `/find` + карта).
2. **Голос V1:** Web Speech + правка текста — за 1 день проверить сценарий в зале.
3. **Голос V2:** серверный Whisper для стабильного RU в prod.
4. **TTS:** встроенный `speechSynthesis` для коротких фраз — дёшево и достаточно.
5. **Расширения без LLM:** POI в индексе, фильтр этажа, `/nearby`.
6. **LLM в чате** — только когда появятся реальные многошаговые диалоги (уровень 3).

Голос **не меняет** ядро поиска — только способ ввода. Не усложнять: один STT-пайплайн, один `find`.

- [ ] Нужен ли объединённый `/find-voice` или два вызова (transcribe + find)?
