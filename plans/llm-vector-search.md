# План: ИИ-помощник для навигации по карте ТЦ

## Цель

Посетитель пишет на естественном языке — *«блин где найти майку»*, *«где круассан»*, *«кофе»* — и получает **список мест в этом ТЦ** с подсветкой на 3D-карте.

Помощник **не знает ничего о мире** — только то, что админы внесли: названия магазинов, теги (`майка`, `одежда`, `круассан`), категории. Вся «умность» — в сопоставлении смысла запроса с тегами зон, а не в большой языковой модели.

**Не в scope:** общие знания, диалог «как в ChatGPT», поиск по SQL/полнотексту.

---

## Упрощённый сценарий (рекомендуемый MVP)

```
«блин где найти майку»
        │
        ▼
  embedding запроса
        │
        ▼
  pgvector: топ зон по смыслу
        │
        ▼
  список: H&M (1 этаж), Zara (2 этаж), …
        │
        ▼
  подсветка на карте
```

**LLM для этого не нужна.** Достаточно embedding-модели + pgvector.

| Что делает «ИИ» | Как |
|---------------|-----|
| Понимает *«майку»* ≈ *«футболка»* ≈ *«одежда»* | Embeddings (семантика) |
| Не выдумывает магазины | Результат = только зоны из БД с score > порог |
| Работает с сленгом и опечатками | Частично embeddings; сильно зависит от тегов |

Текст ответа можно собрать шаблоном на фронте: *«Нашёл 3 места»* + карточки магазинов. Без генерации текста моделью.

### Когда LLM всё же понадобится (позже, не в MVP)

| Запрос | Без LLM | С LLM |
|--------|---------|-------|
| *«майка»* | ✅ список зон | избыточно |
| *«где поесть»* | ✅ если есть теги `еда`, `ресторан` | избыточно |
| *«а подешевле?»* (уточнение в диалоге) | ❌ нужна история + рассуждение | нужна |
| *«что рядом с Zara?»* (география) | ❌ нужна логика расстояний | нужна или отдельный код |

**Вывод:** для *«сказал → получил список мест»* задача сводится к **семантическому поиску по тегам**, а не к чат-боту. Это в 2–3 раза проще по разработке и в 5–10 раз дешевле по железу.

## Текущее состояние проекта

| Слой | Что есть |
|------|----------|
| **Frontend** | Vue 3, 3D-карта (`MallMap3D.vue`), админка для зон |
| **Backend** | FastAPI на Render (`api/`), обработка планов этажей |
| **БД** | Supabase: `malls`, `floors`; зоны хранятся в `floors.floor_json` (JSONB) |
| **Зона** | `id`, `name`, `category` (`shop`), геометрия (`points`), размеры |
| **Теги / ИИ-помощник** | Нет |

Зоны сейчас живут внутри JSON этажа, а не в отдельной таблице. Для retrieval это важно: либо нормализуем данные, либо индексируем JSON при каждом сохранении.

## Пользовательские сценарии

### Админ
1. Выбирает зону на карте.
2. Задаёт название («Paul», «Кофе Хаус»), категорию (`cafe`, `shop`, `service`…).
3. Добавляет теги: `круассан`, `выпечка`, `кофе`, `завтрак`.
4. Опционально — короткое описание: *«Французская пекарня, круассаны и macarons»*.
5. Сохраняет → данные попадают в БД, обновляется векторный индекс.

### Посетитель
1. Открывает строку ввода / чат на карте.
2. Пишет как угодно: *«блин где майка»*, *«круассан»*, *«кофе с собой»*.
3. Видит список мест (название, этаж) — сразу подсвечены на карте.
4. Клик по месту → камера переходит к зоне, при необходимости смена этажа.

## Архитектура MVP (embeddings only)

```
┌─────────────────┐    POST /assistant/find     ┌──────────────┐
│  Vue            │ ──────────────────────────► │  FastAPI     │
│  ввод + 3D-карта│ ◄────────────────────────── │  api/        │
└────────┬────────┘   zones[] + scores           └──────┬───────┘
         │                                              │
         │  highlight / flyTo / switchFloor             │
         ▼                                              ▼
   MallMap3D.vue                              ┌─────────────────┐
                                              │  1. Embed query │
                                              │  2. pgvector    │
                                              │  3. filter score│
                                              └────────┬────────┘
                                                       │
                              ┌────────────────────────┴────────────────────────┐
                              ▼                                                 ▼
                      ┌──────────────┐                                  ┌──────────────┐
                      │  Supabase    │                                  │  Embedding   │
                      │  pgvector    │                                  │  bge-m3 /    │
                      │  (теги зон)  │                                  │  nomic-embed │
                      └──────────────┘                                  └──────────────┘
```

### Поток запроса

1. Пользователь: *«блин где найти майку»*
2. API эмбеддит запрос → вектор `q` (сленг и лишние слова не мешают — embedding схватывает суть).
3. pgvector: `ORDER BY embedding <=> q LIMIT 10`, фильтр `mall_id`, `score > 0.65`.
4. API возвращает список зон: `{ name, floor_label, zone_id, floor_id, score, tags }`.
5. Frontend показывает список + подсвечивает зоны на карте. Текст — шаблон: *«Нашли 4 места»*.

Если ничего не найдено (все score ниже порога) → *«Не нашли. Попробуйте: одежда, еда, кофе…»* — тоже без LLM.

### Архитектура уровня 2 (если понадобится диалог)

Тот же retrieval, но перед ответом — маленькая LLM, которая **только переформулирует уже найденный список**, не добавляя магазинов.

### Поток индексации (при сохранении зоны админом)

1. Собрать текст для эмбеддинга:
   ```
   {name} | {category} | теги: круассан, выпечка | {description}
   ```
2. Вызвать embedding API → вектор.
3. Upsert в таблицу `zone_search_index` (или колонку `embedding` у зоны).

## Изменения в модели данных

### Вариант A — нормализация (предпочтительно)

Вынести зоны из `floor_json` в отдельную таблицу:

```sql
-- zones (магазины/кафе на этаже)
zones (
  id            uuid PK,
  floor_id      uuid FK → floors,
  external_id   text,          -- id из floor_json для связи с 3D
  name          text,
  category      text,
  description   text,
  tags          text[],        -- {'круассан','выпечка','кофе'}
  geometry      jsonb,         -- points, offset, size…
  embedding     vector(1536),  -- pgvector
  updated_at    timestamptz
)
```

**Плюсы:** простые запросы, индексы, RLS, один источник правды для помощника.  
**Минусы:** рефакторинг сохранения этажа (синхронизация `floor_json` ↔ `zones`).

### Вариант B — без рефакторинга (быстрый старт)

Оставить зоны в `floor_json`, добавить отдельную таблицу-индекс:

```sql
zone_embeddings (
  mall_id     uuid,
  floor_id    uuid,
  zone_id     text,           -- id из JSON
  search_text text,           -- склеенный текст для отладки
  tags        text[],
  embedding   vector(1536),
  PRIMARY KEY (floor_id, zone_id)
)
```

При сохранении этажа API пересобирает embeddings для всех зон этажа.

**Плюсы:** минимум изменений в 3D-части.  
**Минусы:** дублирование, риск рассинхрона.

### Теги: свободный ввод vs справочник

| Подход | Когда |
|--------|-------|
| Свободные теги (админ пишет что угодно) | Быстрый старт, мало магазинов |
| Справочник тегов + автодополнение | Когда тегов много, нужна единообразность |
| Гибрид: свободный ввод + предложения из частых тегов | **Рекомендуется** |

## Выбор технологий

### Векторная БД

| Вариант | Плюсы | Минусы | Стоимость |
|---------|-------|--------|-----------|
| **pgvector в Supabase** | Уже есть Postgres, один провайдер, SQL | Лимиты на free tier | $0–25/мес |
| Qdrant Cloud | Специализированный, быстрый | Ещё один сервис | $0–25/мес |
| Pinecone | Managed, простой API | Отдельный аккаунт, vendor lock-in | от $0 |

**Рекомендация: pgvector в Supabase** — для одного ТЦ (50–300 зон) более чем достаточно.

### Embedding-модель

| Модель | Размерность | Качество RU | Где крутить |
|--------|-------------|-------------|-------------|
| `text-embedding-3-small` (OpenAI) | 1536 | Хорошее | Облако |
| **`bge-m3`** (Ollama) | 1024 | **Хорошее** | Локально, ~1.5 GB |
| `nomic-embed-text` (Ollama) | 768 | Среднее | Локально, ~0.5 GB, CPU OK |
| `multilingual-e5-small` | 384 | Приемлемое | Локально, минимум ресурсов |

Для русского при локальном хостинге — **`bge-m3`**. Облако — OpenAI `text-embedding-3-small` без своего GPU.

### LLM (только уровень 2, не для MVP)

| Вариант | Когда | Где |
|---------|-------|-----|
| **Не нужна** | Список мест + шаблон текста | MVP |
| `qwen2.5:3b` (Ollama) | Уточнения в диалоге | Локально, 4 GB RAM |
| GPT-4o-mini | Красивые ответы в облаке | Если не хостите сами |

Подробные таблицы железа — в разделе «Локальный хостинг».

| Сценарий | Сумма |
|----------|-------|

## Локальный хостинг через Ollama

Для MVP **нужна только embedding-модель** — LLM можно не ставить вообще.

```
FastAPI  ──►  Ollama :11434
                └── embed:  bge-m3 / nomic-embed-text
                     │
                pgvector (Supabase)
```

Ollama с одной embedding-моделью весит **< 1.5 GB** и работает на любом мини-ПК без GPU.

### Какие модели подойдут

Для MVP нужна **только embedding-модель**. LLM — см. уровень 2 в конце раздела.

#### Embeddings (единственное обязательное)

| Модель Ollama | Размер | VRAM/RAM | RU | Размерность | Комментарий |
|---------------|--------|----------|-----|-------------|-------------|
| **`nomic-embed-text`** | ~274 MB | ~0.5 GB | Среднее | 768 | Дефолт Ollama, быстро на CPU, хватает для сотен зон |
| **`bge-m3`** | ~1.2 GB | ~1.5 GB | **Хорошее** | 1024 | Лучший выбор для мультиязычности (RU + EN) |
| **`qwen3-embedding:0.6b`** | ~640 MB | ~1 GB | Хорошее | 1024 | Компромисс качество/размер |
| `mxbai-embed-large` | ~670 MB | ~1 GB | Слабее на RU | 1024 | Больше заточен под EN |

**Рекомендация:** `bge-m3` если есть GPU или 16 GB RAM; `nomic-embed-text` если железо слабое (киоск на CPU).

При смене embedding-модели нужно **переиндексировать все зоны** (размерность вектора другая).

#### LLM (только уровень 2 — диалог)

**Для списка мест не нужна.** Если позже захотите уточнения в чате:

| Модель | VRAM/RAM | Когда |
|--------|----------|-------|
| `qwen2.5:3b-instruct` | 4–6 GB | Минимальный диалог на CPU |
| `qwen2.5:7b-instruct` | 6–8 GB | Лучший RU, если нужен чат |

### Сколько ресурсов нужно (MVP — только embeddings)

| Профиль | Железо | Модель | Latency | Стоимость |
|---------|--------|--------|---------|-----------|
| **Минимум** | 4 GB RAM, CPU (N100) | nomic-embed-text | 200–500 ms | $100–200 |
| **Стандарт** | 8 GB RAM, CPU | bge-m3 | 100–300 ms | $150–300 |
| **Централизованно на 10+ ТЦ** | 8 GB RAM VPS | bge-m3 | 100–300 ms | $5–20/мес VPS |

GPU **не нужен** для MVP. Один embedding-запрос — миллисекунды–сотни миллисекунд.

#### Потребление под нагрузку одного ТЦ

| Параметр | Значение |
|----------|----------|
| Зон в индексе | 50–300 |
| Embedding-индекс в RAM | < 5 MB |
| Запросов в час (пик) | 5–30 |
| Нагрузка на CPU | Почти нулевая |

**Вывод:** для MVP хватит любого мини-ПК или того же VPS что FastAPI. Электричество ~$2–5/мес при 24/7.

#### Диск (MVP)

| Что | Размер |
|-----|--------|
| Ollama + embedding-модель | ~0.3–1.5 GB |
| OS | ~15 GB |
| **Итого** | SSD **32 GB** хватит |

### Стоимость локально vs облако (MVP)

| | Облако (embed only) | Локально (embed only) |
|--|---------------------|------------------------|
| Ежемесячно | $0–1 API | $0 API + $2–5 электричество |

| Latency | 50–200 ms | 100–500 ms на CPU |

## Централизация vs свой Ollama на каждый ТЦ

### Варианты развёртывания

```
A) Централизованно                    B) На каждый ТЦ              C) Гибрид
┌─────────────────────┐              ┌──────┐  ┌──────┐          ┌─────────────┐
│ Ollama (1 сервер)   │              │Ollama│  │Ollama│          │ Ollama cloud│
│ FastAPI             │              │+ API │  │+ API │          │ + edge cache│
│ Supabase (все ТЦ)   │              │ ТЦ 1 │  │ ТЦ 2 │          │ киоск → кэш │
└─────────┬───────────┘              └──┬───┘  └──┬───┘          └──────┬──────┘
          │                             │         │                       │
    ТЦ₁ ТЦ₂ ТЦ₃…                        ТЦ₁      ТЦ₂              ТЦ₁ ТЦ₂ (offline OK)
```

### Сравнение

| Критерий | Централизованно (A) | На каждый ТЦ (B) | Гибрид (C) |
|----------|---------------------|------------------|------------|
| **Стоимость железа** | 1× GPU-сервер на все ТЦ | N× мини-ПК ($300–700 × N) | Сервер + дешёвые киоски |
| **Стоимость обслуживания** | Один инстанс, один деплой | N машин, обновления на объекте | Средняя |
| **Качество модели** | Можно 7B–14B на одном GPU | Обычно 3B–7B из-за бюджета | 7B в облаке, 3B fallback на киоске |
| **Latency** | +50–200 ms сеть до сервера | < 100 ms локально | Локально при кэше |
| **Офлайн / интернет** | Зависит от канала | Работает автономно | Частично |
| **Приватность** | Данные на вашем сервере | Данные не покидают ТЦ | Гибко |
| **Масштаб 1→10 ТЦ** | Тривиально (`mall_id`) | Линейные расходы | Оптимален при росте |
| **Пиковая нагрузка** | Очередь на одном GPU | Изолирована по ТЦ | — |

### Нагрузка: хватит ли одного сервера?

Для MVP (только embeddings) один VPS за **$5–10/мес** спокойно тянет **50+ ТЦ**. Embedding-запрос — ~100 ms, нагрузка мизерная.

Данные уже централизованы в Supabase. **Централизовать inference** — очевидный выбор; per-ТЦ железо нужно только для офлайн-киоска.

### Когда что выбирать

| Ситуация | Рекомендация |
|----------|--------------|
| 1–3 ТЦ, есть VPS/сервер | **Централизация (A)** — дешевле и проще |
| 5–20 ТЦ, один дата-центр | **Централизация (A)** — один GPU, фильтр по `mall_id` |
| Киоск без надёжного интернета | **Локально на ТЦ (B)** или гибрид с кэшем |
| Требование «данные не уходят из здания» | **Локально (B)** + локальный Postgres/pgvector |
| Пилот / GREEN CITY один | Облачный embed API или Ollama на том же VPS — **без GPU** |
| Много ТЦ, разные города | **Централизованно** с региональным сервером; per-ТЦ только для офлайн-точек |

### Рекомендация для mall-map

**Оптимальный путь при росте:**

1. **Сейчас (1 ТЦ):** облачный embedding API или Ollama `bge-m3` на VPS — без LLM, без GPU.
2. **2–10 ТЦ:** тот же центральный сервер, фильтр по `mall_id`.
3. **Офлайн-киоск:** мини-ПК с `nomic-embed-text` на CPU + кэш embeddings.

Централизация выигрывает по **стоимости, качеству модели и простоте обновлений**. Per-ТЦ оправдан только для **офлайна, изоляции сети или жёстких требований безопасности**.

## Риски и как снизить

| Риск | Митигация |
|------|-----------|
| Пустые / плохие теги | Подсказки в админке, шаблоны категорий («одежда» → майки, футболки, джинсы) |
| Выдуманные магазины | Нет LLM в MVP — результат только из pgvector |
| Рассинхрон JSON и индекса | Индексация только через API при save |
| Запрос не матчится | Порог score + «попробуйте: одежда, еда, кофе…» |
| Мультиязычность | `bge-m3` или OpenAI embed |

## Рекомендуемый путь

1. **MVP = embeddings + pgvector**, без LLM. Запрос → список мест → карта.
2. **pgvector в Supabase**, таблица-индекс (вариант B), теги в админке.
3. **Embedding:** `bge-m3` (локально) или OpenAI embed (облако) — централизованно на все ТЦ.
4. Текст ответа — шаблон на фронте, не генерация моделью.
5. LLM добавить **только если** понадобится диалог с уточнениями (уровень 2).
